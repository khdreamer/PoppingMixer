Detailed Description

The easy audio input (e.g. microphone) and output (e.g. headphones) library.

Overview

The Little Endian AudioIO library aims to provide an easy way into cross-platform audio input and output with convenient integration with core Little Endian SDKs as well as related native OS facilities.
This is a rather ambitious undertaking and the first release provides only the basic building blocks that future releases will further expand and build upon.
The sample app contains the audioIOSWBridge.hpp header file with basic examples on how to integrate AudioIO and the SpectrumWorx SDK.

General usage and design notes

Requirements

Tools and environment

  • Android:
    • API level 9
    • NDK r8c or later
    • GCC 4.6 or later or Clang 3 or later
    • x86 (SSE3 or later) or ARM (v5TE or later, ARMv7 highly recommended) CPU
    • does not itself link with any STL
    • requires the RECORD_AUDIO and MODIFY_AUDIO_SETTINGS permissions
  • iOS:
    • iOS 5.1.1 or later
    • Clang 3 or later
    • ARMv7 or later CPU
    • links with libc++
    • requires linking with the Accelerate, AudioToolbox and CoreAudio frameworks (the development build additionally requires the Foundation framework)
  • OS X:
    • OS X 10.6 or later
    • Clang 3 or later
    • x86 SSE3 capable CPU
    • links with libstdc++
    • requires linking with the Accelerate and CoreAudio frameworks (the development build additionally requires the CoreServices framework)
  • Windows:
    • Windows Vista or later
    • Visual Studio 2010 or later (express editions are fine)
    • x86 SSE2 capable CPU
    • links with the static MSVC runtime

3rd party libraries

  • Boost 1.50 or later (a header-only subset is required, the needed portion of version 1.55 is bundled with this release).

Build configuration

  • Two builds of the library are provided: the "release" build intended for public release/retail/RTM builds, and the "development" build that has basic sanity checks compiled in that will show assertion failures in cases of incorrect usage (depending on the target OS, assertion failures will be reported with one or more of the following: message boxes, stderr messages, system log entries and aborting or breaking into the debugger)
  • Code specifically optimized for a given architecture/CPU is provided in the form of separate libraries, no built in runtime detection and dispatching is provided. The following builds are provided:
    • Android:
      • ARMv5TE_soft-float (default/minimum guaranteed armeabi architecture)
      • ARMv6_VFP2
      • ARMv7a_VFP3-D16 (default/minimum guaranteed armeabi-v7a architecture)
      • ARMv7a_VFP3-D32
      • ARMv7a_NEON
      • x86-32_SSE3
    • iOS: single universal fat static library with armv7, armv7s, arm64, i386 (simulator) and x86_64 (simulator) slices
    • OSX:
      • x86-32_SSE3 + x86-64_SSSE3 universal static library
      • x86-32_SSE4.1 + x86-64_SSE4.1 universal static library
    • Windows:
      • x86-32_SSE2
      • x86-32_SSE4.1
      • x86-64_SSE3
      • x86-64_SSE4.1
    • for additional support and/or optimizations for specific architectures please contact us.

Error handling

  • Operations that can potentially fail are grouped and isolated in order to offer the no-fail guarantee wherever possible thereby simplifying error reporting. Exceptions are not used and, where required, errors are reported with statically allocated error message strings (denoted by the error_msg_t data type).
  • The library does not handle incorrect usage. Incorrect usage or failure to comply with documented preconditions for a given function results in undefined behaviour or assertion failures in release or development builds, respectively.
