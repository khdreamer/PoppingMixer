Introduction

How to use this document?

Overview

SpectrumWorx software development kits enable easy sound effects creation using a flexible C++ API.

Quick start

For a quick start and usage examples see the Quick start and Tutorial pages.

GUI tool

For instructions on how to design effects using a GUI tool visit the GUI tool page.

Sound effect API

For detailed information on the available effects see Modules. (Please, note that the set of included effects may depend on your license.)

Audio IO

To simplify capture of audio from a microphone or playback through speakers or reading or writing audio files check out the AudioIO helper library.

Requirements

  • tools and environment:
    • Android:
      • API level 8
      • NDK r8c or later
      • GCC 4.6 or later or Clang 3 or later
      • x86 (SSE3 or later) or ARM (v5TE or later, ARMv7 highly recommended) CPU
      • does not itself link with any STL
      • AudioIO requires the RECORD_AUDIO and MODIFY_AUDIO_SETTINGS permissions
    • iOS:
      • iOS 5.1.1 or later
      • Clang 3 or later
      • ARMv7 or later CPU
      • links with libc++
      • requires linking with the Accelerate framework (the development build additionally requires the Foundation framework)
    • OS X:
      • OS X 10.6 or later
      • Clang 3 or later
      • x86 SSE3 capable CPU
      • links with libstdc++
      • requires linking with the Accelerate framework (the development build additionally requires the CoreServices framework)
    • Windows:
      • Windows XP SP3 or later
      • Visual Studio 2010 or later (express editions are fine)
      • x86 SSE2 capable CPU
      • links with the static MSVC runtime
  • 3rd party libraries:
    • Boost 1.50 or later (a header-only subset is required, the needed portion of version 1.55 is bundled with the SDK and is located at /include/boost).

Build configuration

  • The sw_sdk directory has to be in the include path.
  • If inclusion of SpectrumWorx SDK headers with/through the sw_sdk prefix is desired, the parent directory of the sw_sdk directory also has to be included in the include path.
  • Two builds of the SpectrumWorx SDK are provided: the "release" build intended for public release/retail/RTM builds, and the "development" build that has basic sanity checks compiled in that will show assertion failures in cases of incorrect usage (depending on the target OS, assertion failures will be reported with one or more of the following: message boxes, stderr messages, system log entries and aborting or breaking into the debugger)
  • Code specifically optimized for a given architecture/CPU is provided in the form of separate libraries, no built in runtime detection and dispatching is provided. The following builds are provided:
    • Android:
      • ARMv5TE_soft-float (default/minimum guaranteed armeabi architecture, only usable for offline processing)
      • ARMv6_VFP2 (usable in realtime only with simple engine and effect setups)
      • ARMv7a_VFP3-D16 (default/minimum guaranteed armeabi-v7a architecture)
      • ARMv7a_VFP3-D32
      • ARMv7a_NEON
      • x86-32_SSE3
    • iOS: single universal fat static library with armv7, armv7s, arm64, i386 (simulator) and x86_64 (simulator) slices
    • OSX:
      • x86-32_SSE3 + x86-64_SSSE3 universal static library
      • x86-32_SSE4.1 + x86-64_SSE4.1 universal static library
    • Windows:
      • x86-32_SSE2
      • x86-32_SSE4.1
      • x86-64_SSE3
      • x86-64_SSE4.1
    • for additional support and/or optimizations for specific architectures please contact us.

General usage and design notes

  • All SW SDK objects have implicit shared ownership semantics: created on the heap by static create() factory member functions that return (intrusive) shared pointers (that use reference counters internal to objects themselves).
  • The SW SDK is designed to group and isolate operations that can potentially fail thereby simplifying error reporting. It is completely exception free and does not even use special error codes: in the few required places errors are reported through simple boolean or nullptr values.

2.1.0

  • added:
    • standalone effect support
  • fixed:
    • ModuleChain::empty() member function
    • reintroduced the missing usesSideChannel static data member for the Exploder, PVExploder, Imploder and PVImploder effect classes
    • minor documentation fixes and cleanups
  • changed:
    • BREAKING CHANGE - Related to the standalone effect support the ModuleProcessor class was split into the ChainedModuleProcessor and StandaloneModuleProcessor<> classes. The ChainedModuleProcessor class takes on the previous role of the ModuleProcessor class.
    • BREAKING CHANGE - Unified and clarified ownership semantics: all (user creatable) SW SDK objects are now held by reference counted pointers:
      • this primarily affects modules, that is the Module class template that is no longer used as a thin proxy class that wraps a shared pointer, rather it now represents a module directly and is used through a shared pointer:
        Module<PitchShifter> pitchShifter( createModule<PitchShifter> );
        pitchShifter.set<PitchShifter::Cents>( 30 );
        now becomes
        Module<PitchShifter>::Ptr pPitchShifter( createModule<PitchShifter> );
        pPitchShifter->set<PitchShifter::Cents>( 30 );
      • source that used the (Chained)ModuleProcessorPtr typedef and did not rely on the fact that it was a std::auto_ptr requires no change regarding this change and that class in particular
      • as a convenience, all the public, user creatable classes contain Ptr and CPtr (non-const and const versions of the smart pointer respectively) member typedefs (ChainedModuleProcessor::Ptr, Module<Effect>::Ptr, etc.)
    • BREAKING CHANGE - Removed one redundant directory layer from the effects include path, each effect header now resides directly in the effects directory as opposed in a like-named subdirectory within the effects directory, e.g.:
      #include "sw_sdk/effects/pitch_shifter/pitchShifter.hpp"
      becomes simply
      #include "sw_sdk/effects/pitchShifter.hpp"
